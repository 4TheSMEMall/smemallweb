# BHC Gap Engine — Integration Contract (bhctestt.com → SME Mall)

Scope: this document defines **only** the handoff between bhctestt.com (the BHC assessment platform) and SME Mall (this repo), so the SME Mall Gap Dashboard can be built. It does not cover the service marketplace, execution tracker, rating, or admin portfolio views — those are SME Mall-only and out of scope here.

## 1. Division of responsibility

- **bhctestt.com owns:** the assessment, the 6-pillar Q&A, and the answer-to-gap mapping engine. It generates the gap list **once, at the moment an assessment is submitted**.
- **SME Mall owns:** everything after that — storing gaps, tracking status (`OPEN` → `IN_PROGRESS` → `CLOSED`), the Gap Dashboard UI, and (later) service requests, jobs, and ratings.
- bhctestt.com never needs to know about job status, service providers, or ratings. The relationship is **one-directional**: bhctestt.com → SME Mall, once per submitted assessment.

## 2. What bhctestt.com needs to build

### 2.1 Gap rule table (server-side, bhctestt.com)

For each assessment question, define a rule mapping a specific answer to a gap record. Example (from the reference spec):

| Question | Triggering Answer | gap_title | pillar | priority | uplift_score | service_tag |
|---|---|---|---|---|---|---|
| Are you registered with CAC? | No | Not CAC Registered | compliance | CRITICAL | 18 | cac_filing |
| Do you use bookkeeping software? | No | No bookkeeping system | financial | HIGH | 12 | bookkeeping |
| Do you have a business website? | No | No online presence | customers | HIGH | 9 | web_design |
| Do you have written staff contracts? | No | No HR documentation | people | MEDIUM | 7 | hr_consulting |
| Do you have a written business plan? | No | No growth strategy document | growth | HIGH | 10 | business_planning |
| Do you file annual tax returns? | Sometimes | Inconsistent tax compliance | compliance | HIGH | 8 | legal_consulting |

This table is illustrative — bhctestt.com's team defines the full rule set per pillar. **Pillar IDs must be agreed and fixed** before launch (see Open Question #1 below).

### 2.2 Priority calculation

Computed server-side per gap, not just from the overall score:

- `CRITICAL` — pillar score < 45 OR the gap blocks funding/credit access
- `HIGH` — pillar score 45–59, or any compliance/legal risk regardless of score
- `MEDIUM` — pillar score 60–69, growth-opportunity gaps
- `LOW` — pillar score 70+, minor optimisation

### 2.3 The handoff payload

On assessment submission, bhctestt.com must include a `gaps` array in whatever already notifies SME Mall of a completed assessment (existing webhook/callback, or a new one if none exists today). Shape:

```json
{
  "assessment_id": "uuid",
  "client_email": "owner@business.com",
  "score": 63,
  "band": "DEVELOPING",
  "pillar_scores": { "financial": 52, "operations": 71, "customers": 68, "compliance": 41, "people": 65, "growth": 56 },
  "gaps": [
    {
      "gap_id": "uuid",
      "pillar": "compliance",
      "gap_title": "Not CAC Registered",
      "description": "Your business is not registered with the Corporate Affairs Commission.",
      "priority": "CRITICAL",
      "uplift_score": 18,
      "service_tag": "cac_filing"
    }
  ]
}
```

Field notes:
- `gap_id` — generated by bhctestt.com, globally unique, stable if the same gap persists across re-tests (lets SME Mall recognise "this is the same unresolved gap" rather than creating duplicates).
- `pillar` — must match the agreed pillar ID list (Open Question #1).
- `priority` — one of `CRITICAL | HIGH | MEDIUM | LOW`, computed by bhctestt.com per §2.2.
- `uplift_score` — points this pillar's score would gain if the gap were fixed.
- `service_tag` — a stable enum/string SME Mall uses to route to a service category later. Does not need to resolve to anything yet — just needs to exist and be consistent across gaps so SME Mall can group them.
- `client_email` — used by SME Mall to match the gap list to an existing account (same pattern already used for `/wibg/my-status` style lookups).

### 2.4 Delivery mechanism — two options, pick one

**Option A — Push (preferred):** bhctestt.com includes the `gaps` array directly in its existing "assessment completed" notification to SME Mall (if one already exists for syncing `bhcApi.getHistory()` data). No new endpoint needed on bhctestt.com's side.

**Option B — Pull:** if no such notification exists today, bhctestt.com exposes one new endpoint:

```
GET https://bhctestt.com/api/assessments/{assessment_id}/gaps
Authorization: Bearer <service token>
→ { gaps: [ ...as above... ] }
```

SME Mall calls this once, right after detecting a new completed assessment (we already poll/refetch BHC history on window focus — see `apps/web/src/app/(app)/dashboard/bhc/page.tsx`).

## 3. What SME Mall builds (this repo, once the contract above is live)

- New table: `gaps` (gap_id, business_id, assessment_id, pillar, gap_title, description, priority, uplift_score, service_tag, status, created_at) — `status` defaults to `OPEN` and is mutated entirely within SME Mall as the user requests/closes fixes.
- An ingestion step that consumes the handoff (§2.4) and upserts gap rows, keyed by `gap_id` so re-takes don't duplicate unresolved gaps.
- `GET /api/bhc/gaps` — SME Mall's own endpoint, returns the current user's gap list for the Gap Dashboard.
- The Gap Dashboard screen itself: summary header (score, total gaps, critical count, projected score), gap cards (5 states: open-critical/high/medium, in-progress, closed), filter by pillar/priority, sort by impact/urgency.
- "Fix This" on each card — for this phase, shows a "service marketplace launching soon" placeholder (no service request flow yet).

## 4. Open questions to resolve before bhctestt.com starts building

1. **Pillar ID mismatch.** The reference spec uses 6 pillars: `Financial Health, Operations, Customers, Compliance, People, Growth & Strategy`. SME Mall's current data (via `bhcApi.getHistory()` → `sectionScores`) uses different names: `Financial Management, Business Operations, Market Position, Legal & Compliance, Human Resources, Technology & Innovation`. These need to be reconciled — either bhctestt.com renames/restructures its pillars to match the new spec, or the gap engine maps to the existing pillar names. **Decide before building the rule table**, since every rule in §2.1 is keyed by pillar.
2. **Delivery mechanism.** Confirm whether bhctestt.com already has an outbound notification to SME Mall on assessment completion (Option A) or whether a new pull endpoint is needed (Option B).
3. **Re-test behaviour.** When a user re-takes the BHC test, does bhctestt.com regenerate the full gap list from scratch (overwriting), or only emit *new* gaps not previously seen? Recommendation: always emit the full current gap list; SME Mall's ingestion step closes any previously-open gap that no longer appears, and upserts the rest by `gap_id`.
